# SoulVault Glossary

## Agent
A node/process participating in a swarm. Each agent has its own cryptographic identity (keypair). Under ERC-8004 Model 1, an agent may also have its own public ERC-8004 identity record.

## Swarm
A coordination group represented by one smart contract. Membership, epochs, and backup pointers are scoped per swarm.

## Owner
Root authority wallet for swarm initialization, join approvals, rekey operations, and historical key grants. Holds the owner escrow key.

## Join Request
Onchain request from a candidate agent to become an approved swarm member. Includes the agent's raw `pubkey` in calldata so the owner can wrap epoch keys without any offchain storage dependency.

## Approved Member
Agent address recognized by the swarm contract as active.

## Epoch
A membership/version period. Every epoch has a single symmetric content key (`K_epoch`).

## K_epoch
The symmetric key used to encrypt all swarm content for a specific epoch — backups, message payloads, and manifests. Used directly (not derived) in MVP. Shared by all approved members (swarm-readable model).

## Rekey / Epoch Rotation
Generating a new `K_epoch+1` when membership changes (join/kick/manual rotate) so only current members can read future content. Triggered by the owner via CLI (`soulvault epoch rotate`).

## membershipVersion
A uint64 counter on the swarm contract that increments on every join approval or member removal. Used as an optimistic concurrency lock in `rotateEpoch` — the owner's rekey transaction must supply the `expectedMembershipVersion` it prepared against; if membership changed in the interim, the transaction reverts and the owner retries.

## Wrapped Key
`K_epoch` encrypted to a specific member's public key. Only that member can unwrap it with their private key.

## Wrapped Key Bundle
A collection of wrapped keys (one per approved member for a given epoch) plus an `ownerEscrowEntry`, stored as an offchain encrypted object. Referenced onchain by storage reference/hash via `EpochRotated`.

## ownerEscrowEntry
An entry in every wrapped-key bundle containing `K_epoch` encrypted to the owner's public key. Enables the owner to recover any historical epoch key for use in recovery flows and historical key grants.

## Historical Key Bundle
A collection of per-epoch wrapped key entries issued to a specific member covering epochs prior to their join (or after a key loss). Generated by the owner, uploaded to offchain encrypted storage, referenced by `HistoricalKeyBundleGranted` event. Enables new joiners and recovered nodes to decrypt past backups and messages.

## Historical Key Grant
The process by which the owner re-wraps historical `K_epoch` values (recovered via owner escrow) for a new or recovered member's pubkey. Initiated via `soulvault keygrant --member <address> --from-epoch <N>`.

## Unwrap
Local decryption of a member's wrapped-key entry to recover the plaintext `K_epoch`.

## Manifest
Depending on context:
- **backup manifest**: integrity metadata for a backup (file list, per-file hashes, archive hash, epoch, and encryption metadata)
- **agent environment manifest**: public or private capability metadata for an agent

## Backup Bundle
Encrypted archive containing agent continuity files (`SOUL.md`, `MEMORY.md`, `HEARTBEAT.md`, etc.), encrypted with `K_epoch`.

## Pointer Update
Onchain update that records latest backup references (for example storage locator, merkle root, publish transaction hash, manifest hash, epoch) without exposing plaintext.

## File Mapping
The per-member backup publication record written to a swarm contract after a backup run. It links the uploaded encrypted artifact to onchain verification fields such as storage locator, merkle root, publish transaction hash, and manifest hash.

## Message Bus (Event-Driven)
Contract events used as verifiable coordination signals (`JoinApproved`, `EpochRotated`, `AgentMessagePosted`, etc.).

## AgentMessagePosted
Event containing metadata and a reference to an encrypted message payload stored offchain. The payload is encrypted with current `K_epoch`; all swarm members can decrypt it.

## AgentManifestUpdated
Event emitted when an agent publishes or updates its environment manifest (CPU/GPU/RAM/tooling/availability). Manifest is stored offchain; contract stores only the storage reference/hash pointer. A public-safe version may also be referenced from the agent's ERC-8004 `agentURI`.

## AAD (Additional Authenticated Data)
Metadata authenticated during encryption (e.g., `from/to/topic/seq/epoch`) to prevent tampering or message mixups.

## Sequence Number (seq)
Monotonic counter per sender used in message posting to enforce order and prevent per-sender replay attacks.

## TTL
Time-to-live metadata field on messages indicating validity window.

## Member File Mapping
The explicit per-member onchain backup record used in Option B. It stores verifiable backup publication metadata such as `storageLocator`, `merkleRoot`, `publishTxHash`, `manifestHash`, and `epoch` for a given member in a given swarm.

## ERC-8004 (EIP-8004)
An Ethereum draft standard for trustless agents built around Identity, Reputation, and Validation registries. In SoulVault, it is used as an optional **public identity/discovery layer** for individual agents.

## agentURI
The ERC-8004 registration URI for an agent. In SoulVault MVP it is a base64-encoded `data:` URI containing the public registration JSON directly. The registration may include custom SoulVault metadata such as `harness`.

## Harness
A SoulVault convention describing the agent runtime/framework, such as `openclaw`, `hermes`, or another execution stack. This is not a core ERC-8004 field, but can be published in custom registration metadata and used to select the correct backup command.

## Treasury (SoulVaultTreasury)
An org-scoped payable contract on 0G Galileo that holds native value and releases funds on approved fund requests. One per organization per chain. Discoverable via ENSIP-11 multichain `addr` record on the org's ENS name (`coinType = 0x80000000 | chainId`). The treasury verifies mutual consent (`swarm.treasury() == address(this)`) before any release.

## Fund Request
An on-chain request filed by an active swarm member for native value from the org's treasury. The swarm contract enforces membership; the treasury enforces funds custody. Lifecycle: Pending → Approved / Rejected / Cancelled.

## ENSIP-11 / Multichain Address
An ENS standard for storing addresses on multiple chains under a single ENS name. Treasury discovery uses `addr(node, coinType)` where `coinType = 0x80000000 | chainId`. An org with treasuries on multiple chains gets one slot per chain.

## coinType
The ENSIP-11 key used to index multichain addresses. For EVM chains: `0x80000000 | chainId` (unsigned). For 0G Galileo (chain 16602): `coinType = 2147500186`.

## Self-Hosted Mode
User runs their own infrastructure components (e.g., relay/control-plane, optional local storage/indexing services).

## Managed Mode / SaaS Path
Hosted service provides operational components (e.g., relay/control-plane, storage indexing/mirroring) while the protocol remains open-source and portable.

## RekeyRequested
A post-MVP event emitted by the public `requestRekey()` contract function, typically triggered by Chainlink Automation, signaling the owner that a rekey is overdue. The owner CLI responds by initiating the actual rekey.
